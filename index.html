Business Module Analysis

While the inheritance tree shows *what* objects are, the aggregation mapping (HAS-A relationships) shows *how* they interact to form business modules. This section breaks down the architecture by operational domain, illustrating the dependencies required to run the pharmacy.

👥

Human Resources & CRM Module

This module handles all individuals interacting with the system. The foundational Person class establishes core identity, which is then extended into specific roles. Crucially, this module utilizes Composition/Aggregation by attaching an Address object to every person.

Base Identity

  • • Person (Abstract)
  • • Address (Aggregated)

Staff Roles

  • • Admin (IS-A Staff)
  • • Cashier (IS-A Staff)

Customer Tiers

  • • Gold (IS-A Member)
  • • Silver (IS-A Member)
💊

Inventory Management Module

The inventory is divided into consumable medicines and reusable/purchasable equipment. The Equipment class introduces a dependency by utilizing the standalone Promotion utility class to dynamically calculate discounted prices.

Medicines

  • • LiquidMedicine
  • • TabletCapsule
  • Inherits from base Medicine class.

Equipment

  • • FirstAid
  • • Diagnostic
  • Uses: Promotion (Dependency)
🛒

Sales & Transaction Module

The most complex module heavily relies on Aggregation. A Payment acts as the central hub linking the Cashier (Staff), the Customer (Member), and the purchased Goods (Order). The Order aggregates Items, which in turn aggregate specific Inventory components.

Receipt ---(proves)--> Payment
├── (processed by) Staff
├── (paid by) Member
└── (pays for) Order
└── (contains) Item
├── (includes) Medicine
└── (includes) Equipment
